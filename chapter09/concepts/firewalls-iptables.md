# Firewalls & `iptables` 101 (Section 9.25)

Every router—and every Linux server—needs a firewall to ensure that malicious traffic is dropped before it can interact with the system or internal network. In Linux, this packet filtering happens directly inside the kernel using a framework called `netfilter`, which is managed using the `iptables` command.

*(Note: As you've seen in your previous SRE labs, `iptables` is also responsible for NAT and port redirection, but here we are focusing strictly on its role as a security firewall).*

## 1. The Three Main Chains (The `filter` Table)
When a packet moves through the Linux networking stack, it passes through "checkpoints" called chains. In the default `filter` table, there are three primary chains:

- **`INPUT`**: Processes packets coming *in* to the host from the network. (e.g., Someone trying to SSH into the machine).
- **`OUTPUT`**: Processes packets generated by the host going *out* to the network. (e.g., The machine curling an external API).
- **`FORWARD`**: Processes packets being routed *through* the machine to another subnet. (Used only if the machine is acting as a router/NAT gateway).

## 2. Default Policies & The "Default Deny" Strategy
Every chain has a **default policy**—the action it takes if a packet doesn't match any specific rule.
- `ACCEPT`: Let the packet through.
- `DROP`: Silently discard the packet.

**The SRE Best Practice:** You cannot have serious security if you leave the policy as `ACCEPT` and play "whack-a-mole" trying to block bad IP addresses. You must switch to a **Default Deny** strategy.

```bash
# Danger: Ensure you don't lock yourself out before running this!
iptables -P INPUT DROP
```
Once the policy is `DROP`, the machine ignores everything. You must then explicitly poke holes to `ACCEPT` only the exact traffic you want.

## 3. Essential Firewall Rules
When building a "Default Deny" firewall, you must append (`-A`) or insert (`-I`) specific `ACCEPT` rules.

**1. Allow Loopback Traffic:**
Local applications (like a web server talking to a local database) use the loopback interface (`127.0.0.1`). If you don't allow this, internal services will break.
```bash
iptables -A INPUT -s 127.0.0.1 -j ACCEPT
```

**2. Allow SSH (So you don't lock yourself out):**
You must explicitly allow incoming TCP connections on port 22.
```bash
iptables -A INPUT -p tcp --destination-port 22 -j ACCEPT
```

**3. Allow Responses to Outbound Traffic:**
If your server curls `google.com`, the response comes back as incoming traffic. If the `INPUT` chain drops it, `curl` will hang. You must allow incoming packets that are part of an already established outbound connection (Stateful Firewalling).
```bash
# Note: The book uses a slightly older syntax; the modern syntax is usually:
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```

## 4. Basic Management Commands
As you recall from your labs, order matters! Rules are evaluated top-to-bottom. If an `ACCEPT` rule is placed below a wildcard `DROP` rule, it will never be triggered.

- **List rules (with line numbers):** `iptables -L --line-numbers -n -v`
- **Append rule (bottom):** `iptables -A INPUT ...`
- **Insert rule (top):** `iptables -I INPUT 1 ...`
- **Delete rule:** `iptables -D INPUT 3`
